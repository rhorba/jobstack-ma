# ISSUES — JobStack.ma



## ISSUE — 2026-07-06 — Minor — Unauthenticated requests return 403 instead of 401
Verified during Sprint 2 e2e check (Story 2.4 role guards): unauthenticated requests to protected endpoints (e.g. GET /api/v1/candidates/me with no token) return HTTP 403 instead of the conventional 401. Role-based denial (wrong role, valid token) also correctly returns 403 — both cases hit the same handler because no custom AuthenticationEntryPoint is configured in Spring Security, so missing auth falls through to the AccessDeniedHandler.
Impact: Low — access is still correctly denied in both cases, this is a convention/DX issue only (frontend can't distinguish "log in" from "you don't have permission" by status code alone).
Fix: Add a custom AuthenticationEntryPoint returning 401 for missing/invalid credentials, keep 403 for valid-but-insufficient-role. Not blocking Sprint 2 ship; backlog for a future auth polish story.

## ISSUE — 2026-07-06 — Fixed — Test-scoped application.properties shadowed main config
While adding CV upload tests, created src/test/resources/application.properties to override cv.storage.path for tests. Maven puts test resources ahead of main resources on the test classpath with same-filename replacement (not merge), so this silently wiped out jwt.secret, DB config, etc. for ALL test classes — all 23 tests failed with "Could not resolve placeholder 'jwt.secret'".
Fix: removed the test-scoped application.properties entirely; CvStorageService now defers directory creation from constructor to store()-call time, so the default /data/cvs path (harmless on Windows since it's never touched unless a test actually uploads a file) doesn't need any override for most test classes. Only CvUploadTests (which does call store()) uses @TestPropertySource(properties = "cv.storage.path=target/test-cvs") scoped to that one class.
Lesson: never add a same-named src/test/resources/application.properties without copying every property the app needs — prefer @TestPropertySource scoped to the test class that needs the override.

## ISSUE — 2026-07-06 — Fixed — 403 on GET /api/v1/candidates/me against live docker stack (not reproduced in MockMvc tests)
Root cause found (not the two leading hypotheses from the original entry — both were wrong): this was never specific to the candidate profile endpoint or to a missing candidate_profiles row. ANY `ResponseStatusException` thrown from controller/service code in the live stack was coming back as an empty 403, regardless of its real status — reproduced directly against the api container for duplicate-email register (should be 409), wrong-password login (should be 401), and self-register-as-ADMIN (should be 400), all returning 403.
Actual mechanism: `ResponseStatusException` resolution goes through a servlet-container `response.sendError(status)` call, which triggers an internal error-dispatch forward to `/error`. Spring Boot's Spring Security auto-config registers the security filter for `DispatcherType.ERROR` as well as `REQUEST`, so `/error` gets re-evaluated by our `SecurityFilterChain`. `/error` didn't match any of the `permitAll` rules (`/api/v1/auth/**`, `/actuator/health`, `/api/v1/jobs/**`), so it fell through to `anyRequest().authenticated()`. With no custom `AuthenticationEntryPoint` configured, Spring Security's fallback `Http403ForbiddenEntryPoint` fired, overwriting the original status with an empty 403. MockMvc tests never caught this because `MockMvc` (without `.build()` webAppContextSetup with full dispatcher error handling, or when asserting only the exception/status directly) doesn't replay the servlet container's error-dispatch mechanism the same way a real servlet container does.
Fix: added `.requestMatchers("/error").permitAll()` to `SecurityConfig`'s `authorizeHttpRequests` chain (first rule, before the other permitAll matchers).
Verified against the rebuilt live stack: fresh register 200, duplicate register 409, wrong-password login 401, self-register-as-ADMIN 400, GET/PUT `/api/v1/candidates/me` 200, cross-role access still correctly 403, CV upload/download 200/200. Full backend suite still 23/23 green after the fix.
Not fixed (separate, pre-existing, still backlog): unauthenticated (no-token) requests still return 403 instead of 401, per the other entry above — that needs an actual custom `AuthenticationEntryPoint`, distinct from this bug.
